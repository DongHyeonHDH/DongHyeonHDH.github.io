# View Resolver
---
컨트롤러가 반환한 객체는 논리적 뷰 이름으로 반환받는데 이를 실제 view로 변환해주는 역할을 수행한다.

- 역할
    - 컨트롤러는 특정 뷰를 식별하기 위해 문자열 형태로 뷰 이름을 반환하게 된다.
    뷰 리졸버는 이를 실제 뷰 템플릿 파일로 매핑을 한다.

# Spring Container(IoC Container)
---
객체 생성과 생명주기를 개발자가 아닌 Spring이 대신 관리해주는 공간
**스프링 컨테이너에 의해 관리되는 재사용 가능한 sw 컴포넌트**인 bean을 생성 이를 통해 의존관계 연결, 관리를 수행한다.

스프링 빈들의 생명 주기를 관리하며, 생성된 자바 객체들에게 추가적인 기능을 제공한다.
스프링 컨테이너를 통해 원하는 만큼 많은 객체를 가질 수가 있다.
의존성 주입을 통해 애플리케이션의 컴포넌트를 관리할 수가 있다.

@Configuration을 클래스에 붙이면 이 클래스를 설정정보를 사용가능하다.
그리고 클래스 내부에 @Bean을 붙인 메서드를 스프링 컨테이너에 등록하는 것이 가능하다.
```java 
@Configuration
public class AppConfig {
	
    @Bean
    public BeanOne beanOne() {
    	return new BeanOne(beanTwo());
    }
```

# 의존성 주입(Dependency Injection)
---
객체가 필요로 하는 다른 객체(의존 객체)를 직접 생성하지 않고, 외부에서 전달받는 설계 방식
```java
public class OrderService {
    private MemberRepository memberRepository;
}
```
만약 여기서 다른 레포지토리를 사용하고 싶다면 코드를 수정해야 한다. 이러한 강하게 결합된 클래스를 분리하고 애플리케이션의 실행시점에 객체 간의 관계를 결정함으로써 결합도를 낮추고
유연성을 확보한다. 이 경우 외부에서 memberRepository를 넣는 것을 외부에 맡기게 된다.

- IoC와의 관계
    IoC가 조금 더 넓은 개념이라고 볼 수가 있다. IoC를 구현하는 방법이라고 볼 수가 있다.

아래는 DI를 구현한 코드이다.

```java
@Service
public class OrderService {

    private final MemberRepository memberRepository;

    public OrderService(MemberRepository memberRepository) {
        this.memberRepository = memberRepository;
    }
}
```

# 제어의 역전(Inversion Of Controll)
---
객체 생성과 실행 흐름의 제어권을 개발자에서 프레임워크로 넘기는 것
개발자가 직접 Controller을 호출하지 않고 Spring이 그 역할을 해주는 것을 의미한다.

이 제어의 역전으로 변화하는 요구사항에 대한 확장성을 넓힐 수 있고 코드의 결합도가 낮아져 테스트 용이성을 증가시킬 수가 있다. 이 제어의 역전의 대표적인 예시는 두가지가 있는데 첫번째는 위에서 언급한 의존성 주입이다. 의존성 주입을 통해 두번째로는 스프링 프레임워크가 있다. 우리가 흔히 쓰는 라이브러리의 제어권은 우리한테 있지만 프레임워크의 제어권은 프레임워크가 가진다. 스프링의 경우 객체 메서드의 호출을 스프링에게 맞길 수가 있어 유지 보수성을 키울 수가 있습니다.

# Servlet
---
웹요청을 처리하는 클래스이다. http request를 java가 알아듣지를 못한다. 그래서 이것을 java 객체로 변환하는 것이 서블릿이라고 볼 수가 있다. 서블릿은 인터페이스 구조로 되어있다.

```java
public class MyServlet extends HttpServlet {

    @Override
    protected void doGet(HttpServletRequest request, HttpServletResponse response) {
        // 요청 처리
    }
}
```

httpServletRequest로 클라이언트의 요청정보를 받고 클라이언트에 보낼응답을 HttpServletResponse 형에 저장을 한다.


### 서블렛의 생명주기
1. init()
2. service()
3. destroy()

> 참고
    하나의 객체를 여러 요청이 공유하기 때문에 멀티스레드 문제를 잘 다루는 것이 중요하다
    그렇지만 사용자가 직접 서블릿을 다루면 코드가 복잡해져 IoC가 필요하다. 이러한 IoC를 반영한 것이 DispatcherServlet이다.

# Servlet Container
---
서블릿을 생성하고 실행하며, http 요청/응답을 관리해주는 서버환경을 의미한다.
대표적인 예시로 Apache Tomcat, Jetty, Undertow가 있는데 Spring boot 환경에서는 Tomcat을 내장하고 있다.

- 통신하는 순서
    1. 클라이언트 요청 발생
    2. Tomcat이 요청 받음
    3. Request/Response 객체 생성
    4. 적절한 Servlet 호출
    5. 결과를 HTTP 응답으로 변환
    6. 클라이언트에게 반환

서블릿 컨테이너는 서버 측에서 동적으로 웹페이지를 생성하기 위해 사용을 하고 서블릿들과 웹서버의 상호작용에서 사용하기 때문에 스프링에서 필수적인 요소라고 보면 된다.
