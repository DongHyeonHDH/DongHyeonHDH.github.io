# MyBatis DynamicQuery 구현
---

## MyBatis if문 구현하기
```java
 <select id="search" resultType="domain.Product">
        SELECT
        id,
        name,
        price,
        stock
        FROM
        product
        WHERE 1=1
        <if test="name != null and name != ''">
            AND name LIKE CONCAT('%', #{name}, '%')
        </if>
        <if test = "minPrice != null and minPrice > 0">
            AND price &gt;= #{minPrice}} 
        </if>
        <if test = "maxPrice != null and maxPrice > 0">
            AND price &lt;= #{maxPrice}}
        </if>

```
여기서 부등호 >는 앵글브라켓일 수도 있고 부등호가 될 수가 있는데 컴퓨터는 이를 비교해주기가 힘들다.
그래서 크고 작고를 사용할 것이면 대체 식을 사용해야 한다,
escaping : 위혐이 될만한 문자열을 다른 문자열로 대체하는 것
>왼쪽이 오른쪽보다 큰,작은 부등호 표시 &gt, &lt
그럼 && 와 같은 and 연산자와 헷갈릴 수 있기 때문에 이에 관한 escaping 문자가 필요하다
> &amp는 & 부등호와 같이 and 연산자를 표시해주는 기능을 수행한다.

## MyBatis for문 구현하기
```xml
<select id="findByIds" resultType="domain.Product">
    SELECT
        <include refid="productColumns"/>
    FROM
    product
    <where>
        <choose>
            <when test="ids != null and ids.size() > 0">
                id IN
                <foreach collection="ids" item="id" open="(" separator="," close=")">
                    #{id}
                </foreach>
            </when>

            <otherwise>
                1 = 0
            </otherwise>
        </choose>
    </where>
</select>
```
여기서 <where>은 MyBatis의 동적 기능을 수행하는 것으로 조건있을 때 AND, OR 정리를 하는 기능이다.
그리고 <choose><when><otherwise>는 java의 swtich - case와 유사하다고 보면 될 것 같다. 
다음과 같은 정렬 기능을 구사하는 경우 enum을 활용해서 하면 좋을 것 같다.

```xml
<select id="findByIds" resultType="domain.Product">
        SELECT
            <include refid="productColumns"/>
        FROM
        product
        <where>
            <choose>
                <when test="ids != null and ids.size() > 0">
                    id IN
                    <foreach collection="ids" item="id" open="(" separator="," close=")">
                        #{id}
                    </foreach>
                </when>

                <otherwise>
                    1 = 0
                </otherwise>
            </choose>
        </where>
    </select>

```

# Logback과 Log이야기
---
## Logback이란?
java에서 자주 사용하는 로깅 구현체 중 하나이다. 비동기 로깅 지원, 내부 최적화, garbage collector 부담이 적어 대용량 트래픽 서버에서도 사용하기도 한다.

#### 자주쓰이는 Logback.xml  설정파일
```XML
<?xml version="1.0" encoding="UTF-8"?>
<configuration>

    <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <logger name="mapper" level="DEBUG" additivity="false">
        <appender-ref ref="STDOUT" />
    </logger>

    <logger name="java.sql.ResultSet" level="DEBUG" />

    <root level="INFO">
        <appender-ref ref="STDOUT" />
    </root>

</configuration>
```
## Log에 관해서
---
LOG에 관련된 정보를 절대절대 노출되지 않게 할 것

실제 개발에 있어서 배포를 할 때 로그 레벨을 debug로 낮추어서 배포를 하는 것은 안된다., Info로 급을 올려서 보여지도록 해야 한다

#### Logback에서의 로그의 분류
---
1. ERROR 
2. WARN
3. INFO
4. Debug
5. TRACE

INFO가 기본으로 설정으로 되어있는 것이고 debug나 trace는 임의로 낮춰야지 볼 수가 있다.

WARN은 문제가 될 수 있으나 넘어가도 되는 경우가 있다. 다만 심각한 문제가 될 수가 있으니 살펴보는 것은 좋다. → 논리적으로는 오류이지만 실제로는 오류가 아닌 경우도 있다.

ERROR : 오류 발생, spring이 종료 될 수가 있다.

FATAL: Error보다 심각한 문제로 실행이 되지 않는다.

# n+1이 나타나는 이유
---
위 1번 코드를 통해 나오는 결과는 1개이지만 실제 터미널에서 행해진 쿼리를 봤을 때 1 + n번이 발생한 것으로 볼 수가 있다. member을 null로 두지 말고 가져오라고 시켰기 때문에 발생하게 된다. **order과 연관된 member을 가져오는데 join 연산을 하지 않아 발생하는 문제라고 볼 수가 있다.**

지금은 약 100개의 row에 대해 실행해서 문제가 없었지만 실무에서는 이것보다 훨씬 데이터가 커져 문제가 크게 될 수가 있다. 

이러한 n+1문제와 관련되서 **lazy loading**이 있는데 lazy loading은 중요도가 떨어지거나 당장 화면에 보이지 않는 요소들의 로딩을 우선적으로 시행하지 않음으로서 웹페이지 로딩 기술을 향상시키는 방법이다. 이러한 lazy loading은 프록시 객체를 사용하여 프록시 객체가 실제 값에 접근시 실체화를 하는데 이는 프록시 객체를 여러번 접근할 수 있기 때문에 n+1 문제를 해결하는데 좋은 방법이 아니다.

다른 해결방안으로는 **fetch join**이 있는데 fetch join은 연관된 엔티티를 join을 해서 한 번의 쿼리로 함께 가져오는 방법이다. 

예를 들어
```java
    select o from Order o join fetch o.member
```
위 쿼리처럼 order가 참조하는 member도 같이 채워서 가져오는 방법이다. 이게 왜 n+1문제를 해결하는 방법이 되냐면 fetch join은 연관 객체를 처음부터 같이 조회를 하여 추가 조회를 하지 않아도 된다. 


#### 그렇다면 lazy는 안좋은 방법인가?
여전히 lazy는 최적화를 위해 필요한 방법이다. 따라서 기본 설계는 Lazy로 두고 필요한 조회에서 fetch join으로 최적화를 하여 n+1문제를 피하는 방향으로 설계를 두는 편이 좋을 것 같다. 다만 fetch join은 연관객체를 항상 같이 들고오므로 들고오는 데이터 양이 많아질 수가 있다. 그래서 최적화가 필요하고 정확도를 중시하는 분야의 몇몇 기업들은 sql mapper을 사용하여 쿼리 튜닝을 해서 n+1문제 발생을 방지한다.
